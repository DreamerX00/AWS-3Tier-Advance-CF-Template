# Chapter 2: Anatomy of a God-Level Template

CloudFormation templates are highly structured. Knowing how to utilize each section efficiently separates juniors from senior architects.

## 1. Parameters (Runtime Inputs)
Parameters make templates reusable across environments (`dev`, `staging`, `prod`). Never hardcode values that change per environment.

**Scale in Practice:** Root.yaml now exposes ~150 parameters covering every configurable aspect of all nested stacks -- from VPC CIDRs to database engine versions to container CPU/memory. This provides full granularity control, allowing operators to customize any dimension of the architecture without modifying the templates themselves.

**Expert Practices:**
*   Always use `AllowedValues` or `AllowedPattern` to fail fast during validation, not during deployment.
*   Use AWS-Specific Parameter Types (e.g., `AWS::EC2::VPC::Id`) so the AWS Console renders a dropdown of existing VPCs instead of a plain text box.

```yaml
Parameters:
  EnvironmentName:
    Type: String
    AllowedValues: [dev, staging, prod]
    Default: dev
    Description: Used to tag resources and name physical instances.
  DatabaseInstanceClass:
    Type: String
    Default: db.t3.micro
    AllowedValues: [db.t3.micro, db.m5.large, db.r5.xlarge]
    Description: Size of the RDS instance.
```

## 2. Resources (The Meat)
The `Resources` section is the only required part of a template. It defines the AWS components.
Every resource requires a `Type` and usually `Properties`.

**The Logical ID vs. Physical ID Concept:**
In the example below, `MyCustomLogGroup` is the **Logical ID**. The actual name in AWS (the **Physical ID**) will be generated by CloudFormation (e.g., `stackname-mycustomloggroup-abc123xyz`) unless you explicitly set the `LogGroupName` property. *Best practice: Let CloudFormation generate the physical names to avoid naming collisions when deploying multiple stacks.*

```yaml
Resources:
  MyCustomLogGroup: # <-- Logical ID
    Type: AWS::Logs::LogGroup
    Properties:
      RetentionInDays: 14
      # LogGroupName: /ecs/my-app # (AVOID THIS: prevents renaming/recreation)
```

## 3. Outputs (The API of your Stack)
Outputs export values from this stack so other stacks (Nested Stacks or Cross-Stack References) can use them.
Treat `Outputs` as the public API of your CloudFormation module.

```yaml
Outputs:
  VpcId:
    Description: The ID of the VPC
    Value: !Ref MyVPC
    Export: # Using Export allows other stacks to !ImportValue this
      Name: !Sub "${EnvironmentName}-VpcId"
```

## 4. Mappings (Static Lookup Tables)
Use `Mappings` to eliminate `if/then/else` spaghetti logic for values that map statically to a key (often the AWS Region or Environment).

A classic use case is mapping AMI IDs to Regions, because AMI IDs are region-specific.

**Real-World Example from This Repo:** The S3 bucket policy template uses an `ELBAccountMap` mapping with 28 AWS region-to-ELB-account-ID entries. This is a prime example of real-world Mappings usage -- the ELB access log account ID is different per region and is a fixed, known value, making it a perfect `!FindInMap` candidate rather than a parameter.

Here is the classic AMI mapping pattern:

```yaml
Mappings:
  RegionMap:
    us-east-1:
      AmazonLinux2: ami-0c02fb55956c7d316
    eu-central-1:
      AmazonLinux2: ami-07dfba995513840b5

Resources:
  MyEC2Instance:
    Type: AWS::EC2::Instance
    Properties:
      ImageId: !FindInMap [RegionMap, !Ref "AWS::Region", AmazonLinux2]
```

## 5. Conditions (Environment-Aware Toggling)
Conditions allow you to deploy different resources based on the input parameters without duplicating the template.
*Example: Only deploy a NAT Gateway in `prod`, but not in `dev` to save money.*

```yaml
Conditions:
  IsProduction: !Equals [!Ref EnvironmentName, prod]

Resources:
  ProductionNatGateway:
    Type: AWS::EC2::NatGateway
    Condition: IsProduction # Only created if IsProduction is true
    Properties:
      SubnetId: !Ref PublicSubnet1
```
Use `AWS::NoValue` to conditionally omit a specific *property* rather than an entire resource.

**Real-World Condition Examples from This Repo:**
*   `Has3AZs` -- toggles between 2-AZ and 3-AZ deployments, conditionally creating the 3rd AZ's subnets, NAT Gateways, and route tables.
*   `HasHTTPS` -- enables optional TLS listeners and ACM certificate associations on the ALB.
*   `IsFargateFrontend` / `IsFargateBackend` -- switches ECS launch type between Fargate and EC2, conditionally creating the appropriate capacity providers and networking configurations.
*   `IsMySQLOrMariaDB` -- conditionally exports engine-specific CloudWatch log groups (e.g., `audit`, `error`, `slowquery`) based on the selected database engine.

## 🎯 God-Level Checklist for Chapter 2
- [ ] **When would you use `Mappings` instead of `Parameters`?** (Answer: When the relationship is fixed and known beforehand, like Region -> AMI mapping. Parameters are for inputs determined at deployment time.)
- [ ] **Which values should always be in `Outputs`?** (Answer: ARNs, IDs, and Endpoints needed by dependent stacks or external CI/CD pipelines. Do not output sensitive secrets.)
- [ ] **How do you prevent naming collisions between multiple deployments of the same template?** (Answer: Do not explicitly define physical names; let CloudFormation generate them based on the Logical ID and Stack Name.)
